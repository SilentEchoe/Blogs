随着大语言模型能力不断衍进，以LLM为基座的应用或服务爆发性增长，这对模型的推理能力提出了新的挑战：需要同时满足**高吞吐、低延迟**，并实现**内存/显存的高效利用**等关键目标。

这些需求推动了云原生在AI基础设施中的落地，一系列围绕着 Kubernetes 原生构建的高性能分布式推理框架应运而生。这些框架通过更先进的**显存管理**与**请求调度**策略，在 Kubernetes 环境中实现接近硬件上限的推理效率，并进一步提供**GPU 资源调度优化**、**多副本负载均衡**、**模型热加载**与**弹性伸缩**等能力，逐步成为现代模型推理服务的基础设施底座。

vLLM 是一个源于 UC Berkeley 的开源项目，它本身是一个专为高吞吐，低延迟LLM推理设计的系统。它的核心要点在于**分页的KV缓存管理**和**连续批处理机制(Continuous Batching)**，本篇文章主要围绕vLLM处理推理请求的核心展开，会逐一解析其中的关键部分。



### PagedAttention与KV缓存管理

**PagedAttention** 是vLLM的核心创新之一，针对大模型推理中的KV缓存管理提供了类似操作系统虚拟内存的解决方案：Cache Page 按序列整页交换，很好地避免了内存的碎片化和频繁拷贝的开销。

传统 Transformer 推理往往为每个序列预分配最大长度的连续显存空间存储 K/V 矩阵，导致大量未用空间浪费和内存碎片。而 **PagedAttention** 将每个请求的 KV 缓存拆解成固定大小的**块（block）**，每个块可以容纳预定数量的Token的 KV，这些块在显存中可以分散存储不必强制连续，它由块表(block table)统一管理，类似于操作系统将内存分页存储并通过页表映射给进程一样。直接实现按需加载和复用，若多个请求带有相同的前缀则可共享对应的KV缓存页。

通过上述机制KV缓存利用率提升到90%以上，相比传统连续内配的方式只有20%～40%的利用率无疑是极大的提高。









在vLLM中，一个请求相当于是一个“进程”，它的KV缓存由一系列”块“组成，相当于这个进程的虚拟地址。当新的 token产生时，调度器会调用KVCacheManager分配一个空闲块给该token的KV存储，并记录块ID。这些块ID在请求的生命周期中保持不变。

请求无需预先占用一整快连续的显存，在需要的时候再做动态分配，如果有多个请求带着相同前缀则可共享对应的KV缓存页，这在上述机制上更进一步减少分配的频率。



vLLM 针对自回归生成的特点，实现了连续异步批处理，即：不需要等待整个批次所有请求完成后才能处理下一个批次，而是实时将新请求插入正在进行的解码循环中。这种动态批调度让已经完成的生成的请求腾出计算槽位，可以将计算想象成一个进出口水池，计算完毕的Token像水一样流出，新的请求像水流一样填满整个计算槽，这种方式保持了GPU满负荷工作，减少等待和空闲时间。



在调度算法上，vLLM 可以配置**Auto-Batching**策略，根据当前每个序列的进度（如何时需要下一轮前向计算）来决定批次组成，实现真正的流水线并行。为降低CUDA启动开销，vLLM 还运用了 **CUDA Graph** 技术：在初始化时捕获代表不同batch大小的计算图，后续推理直接复用预编译图，从而减少kernel launch overhead，进一步压低了单token生成的延迟 





### GPU 资源调度与并行

**Non-Uniform Memory Access**(NUMA) 在中文中一般称为 非一致性内存访问架构，它描述的是一台多路服务器中，CPU访问不同内存区域的速度并不一样。现代服务器通常情况下是多路CPU，多插槽。当多个CPU抢同一块内存，那么总线就会成为瓶颈。为了避免这种情况，硬件会把内存“分片”挂在不同CPU附近。

于是形成一台服务器中先分成几个 NUMA Node的状态，每个 Node 都有离自己“更近”的CPU,内存，甚至PCle设备。Non-Uniform 本意就是访问本地更快，访问远端内存更慢。

在vLLM 引擎在设计上支持**模型并行和数据并行**等多种并行模式，以扩展模型规模或提升吞吐。在官方的 Quick Start 示例中，可以通过设置`tensor_parallel_size=2`来将模型切分到两块GPU上并行执行。vLLM内部通过 PyTorch Distributed/NCCL 等通信库在设备间同步张量计算结果。如果是多节点部署，vLLM提供了结合 Ray或 kubeRay参考方案。

如果启动 **Pipeline Parallelism with KubeRay** 那么不同节点各负责模型的不同层分段，假设一个60层(神经网络层)的大模型分布在3个节点中，那么每个节点分到20层。每当收到一个新的请求，会按照层数的顺序从高到低：1～20层先计算，当计算完毕后得到一个**中间激活值**，通过 Ray 的远程调用来同步这个消息结果。

整个过程和批处理方式一样，由各个不同阶段的管道组成，Token像水流一般经过其中，最终计算出不同的结果。类同于工厂的流水线，当第一批Token在第二个节点进行计算时，第一个节点也可能正在处理下一批的请求。形成一种重叠执行，以提高整体吞吐量。

vLLM对GPU资源也使用了 NUMA 拓扑。通常情况下每个Worker进程都会绑定一个GPU以及其所属 CPU NUMA节点，以此来减少跨NUMA 内存访问延迟。在 Kubernetes中使用 Topology Manager + CPU Affinity来保证这个行为。

> Topology Manager 是 kubelet 中的一个拓扑对其组件，其作用主要是协调CPU，设备(例如GPU)，内存等资源分配，尽量让它们在拓扑上对其，目标就是降低延迟，提升吞吐。
>
> CPU Affinity 本意是指：将一个任务的可运行CPU集合约束到指定逻辑CPU集合上，也就是可以让一个进程/容器优先或仅允许在某些CPU核上运行。在 Kubernetes 中，这种约束主要通过 **cpuset cgroup controller** 来实现，当kubelet 启用 `static policy`时，**属于 `Guaranteed` QoS 且 CPU `requests` 为整数核的容器**，可以被分配到节点上的 **exclusive CPUs**。所以这些CPU不再分配给其他的Pod共享使用，这个隔离性是强制的。
> 但是这不意味着是**物理机的绝对独占**，容器服务使用的是Linux的Cgoup能力来进行隔离，所以这里的语义更倾向于是针对其他的Pod。

如果在双路服务器上部署 vLLM，多GPU进程应被调度到各自本地的CPU核，以充分利用本地内存带宽。这种 **NUMA-aware** 调度在 Kubernetes 中可通过设置 single-numa-node 策略或 CPU Manager 的 static 模式实现，从而提升 GPU-CPU 通信效率。



除了Ray管理pipeline并行，vLLM还支持KV Cache 跨节点共享，它可以通过一个独立的KV缓存服务(LMCache)实现不同实例之间KV数据的交换或统一存储。















### Scheduler 调度器

Scheduler 负责维护请求队列，它还可以控制每个 Engine 步骤处理那些请求，以及每个请求处理多少Token。vLLM的调度器需要实时感知GPU KV缓存占用情况以及每个请求的状态。

在**Prefill阶段**(需要解释)可以按批处理尽可能多的prompt token,**decode阶段**按序逐步生成。
