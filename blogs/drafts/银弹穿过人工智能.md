纵观整个软件工程历史，人们似乎一直在寻找一种能一劳永逸解决复杂性的方法。

在汇编时代，开发者需要直接为机器执行的每个细节负责。汇编并不会替你管理寄存器、栈、内存以及调用约定，这些都需要开发者亲自维护。这不像现代程序中的“业务逻辑”，而是CPU如何执行的精确描述。

为了屏蔽这类过于底层的复杂性，编程语言和编译器诞生了。

高级编程语言显著降低了表达成本。它使用变量、表达式、函数、条件、循环组成了更强有力的表达，替代了大量繁琐指令拼接与跳转控制；而编译器则接管了原本需要开发者手动维护的底层细节，例如寄存器分配、栈帧组织、参数传递、返回值处理以及指令生成。

随着互联网的兴起，软件规模不断的扩大，应用场景增多。不同的语言围绕着新的问题继续分化和迭代。C++试图保留C的性能与控制力的同时，引入了更强的抽象能力；Go语言则更加关注工程效率，并发模型与大规模服务开发的成本，以简洁和易用得到了开发者的青睐；Rust则是走向另外一个方向，它试图在系统编程中同时兼顾性能、安全与并发，使用强大的编译器来严格要求系统的规范和稳定，随之而来的是陡峭的学习路线;

编程语言并没有消除复杂性，随着系统规模继续膨胀，复杂性从“如何编写单体程序”转变为了“如何组织大规模代码”。在这个阶段**设计模式**流行起来，它的背后是针对于常见设计问题的经验总结，对象如何协作，职责如何划分，扩展如何控制。它作为一种最佳实践，试图解决代码结构和抽象层面的复杂性。

当单机程序进一步演化为大型系统中，复杂度再次涌现。开发者需要考虑的不只是模块、类和接口，而是单体应用中庞大的代码量带来的混乱：模块之间的界限变得模糊不清，代码的维护和Bug的修复变得异常复杂，难以知晓全貌的业务需求，无法追溯的技术选型……

在微服务时代，将一个巨大单体的应用拆分成各个小的服务，这些服务以网络请求聚合在一起，以发挥出不同编程语言的特性，利用其他的编程语言来弥补另外一种编程语言的短板，最终发挥出各自编程语言的长处。

这种组建方式听起来非常美好，但微服务依然不是银弹，即使它在云原生的推动下获得了极大的发展。将庞大的单体结构程序拆分成松耦合的独立组件，如何管理这些服务变成了新的问题。

云服务厂商将独立的计算资源组合起来构成了一个巨大的资源池，软件用户只需要租用计算资源，根据业务的各个阶段来实现计算资源扩缩容。从软件开发到部署发布及运维，可以在云平台迅速实现。监控，回滚，扩缩容，故障恢复，这些原本需要消耗巨大精力维护的东西以租借的方式转嫁到了云厂商。程序在整个开发周期中省略了繁琐的调试/部署。

在Kubernetes已成为云原生的事实标准，无数开发团队标榜自己使用“敏捷开发”，结合CI/CD工具自动化部署，降低运维成本，将基础设施构建的复杂沉入了云端，从而分离分离出来DevOps工程师，SRE等岗位。但也只是对传统软件开发模式进行的迭代和改进。

从宏观来说复杂性隐藏在云端，藏在Kuebrenets的底层细节中，它没有消失。只是由另外一部分开发者来承担。

低代码平台使用各种模版组件，动态设置的方式，以一种胶水的方式生成一个又一个低成本应用，这些应用有些托管在云端，有些私有化部署。中小企业利用这些低成本快速生成的软件来标榜步入数字化企业的行列，以标榜自己并没有落伍。

如今“银弹”终于落到了人工智能身上。**大语言模型当然只是人工智能中的一个分支，但在最近两年里，它几乎被公众等同为了人工智能本身**。无论是模型公司令人咂舌的估值还是一系列重金宣传的Agent应用，已将这个子领域变成了人工智能代名词，深深根植在人们的潜意识中。

从效果上来看真的很**接近**于银弹，甚至类似于魔法，计算机程序的构造和解释(SICP)开篇第一章就将计算过程描绘成控制“精灵”的巫术，在人工智能发展的漫长历史中，终于捧出了一瓶“万灵药”。

在过去两年内，Coding领域被大模型迅速接管，AI生成出来的代码质量有了质的飞跃。这一切看起来像是新的魔法：输入自然语言，输出优质的代码。

> 软件工程就是为了更高质量、更快效率、更低成本的构建“可以运行的软件”。

看起来敏捷宣言已经被大语言模型打破了。通过AI可以将原本月计的工作量缩短到周以内，经受过无数优质代码训练的模型不知疲倦，且可源源不断产出代码，当出现一个包含数万行代码的PR也就不是什么奇怪的事情了。

但是复杂性真的被解决了吗？

在AI Coding的加持下，开发者本身反而成为了最大的瓶颈，从最开始的“半自动化”，即开发者开始Rview Code 每个PR。到最后直接放弃审核，转为“全自动化”Agent审核另外一个Agent所产出的代码，这种无比信任一个黑盒的输出，任由其侵入自己所负责产品的方式真的解决了复杂性吗？

是否只要开发者无需面对软件工程中的复杂性，就意味着找到了银弹？从第一性原理来看，AI并没有消灭复杂性，而是吞噬“显性的复杂性”，同时制造“隐形的复杂性”。LLM依然存在幻觉，在复杂工作中依然会产生不可预料的问题，在复杂繁琐的工作场景中，不可预料的每一步都像是走在钢丝绳上。





> 我发现即使是Codex这样的Coding Agent 依然没办法一次性将事情做对，比如在代码量到了一定程度后对原有的需求进行一系列改动，它就没办法一次性将问题很好地解决
